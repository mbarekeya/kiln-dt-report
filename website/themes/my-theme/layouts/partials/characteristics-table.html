Characteristics Description
C1 System under study An industrial wood drying kiln located at La Doré, Quebec, containing two rail systems for lumber loading, a reversible fan, steam heating coils, ventilation dampers, and sensors for temperature and humidity monitoring.
C2 Physical acting components fan: Circulates hot air throughout the kiln chamber; speed varies by drying phase and direction reverses every 2 simulated hours
door: Sliding door that closes at cycle start and opens manually by operator at cycle end
heater_coils: Steam-fed heating coils that raise dry-bulb temperature progressively from ambient to target Ts across drying phases
dampers: Ventilation dampers that regulate the exchange of moist internal air with external air to control relative humidity inside the chamber
C3 Physical sensing components th_sensor: Wet-bulb temperature sensor (Th), used with Ts to compute relative humidity via psychrometric equations
ts_sensor: Dry-bulb temperature sensor (Ts), measures the air temperature inside the kiln chamber in degrees Celsius
C4 Physical-to-virtual interaction sensor_data_trans: Published every 2 real seconds to RabbitMQ queue wood.kiln.sensor as JSON payload containing Ts, Th, derived RH, estimated MC, current phase, elapsed hours, fan speed, fan direction, and cycle status
phase_data_trans: Current drying phase index and name, air velocity, fan direction, duration remaining, and progress percentage
C5 Virtual-to-physical interaction Operator-triggered: cycle start via RabbitMQ (wood.kiln.start) and stop via HTTP POST /api/stop. Automatic DT control: Ts setpoint adjustment (+-2C) and damper aperture published to wood.kiln.control in response to SLOW_DRYING, FAST_DRYING, and DTAB deviations. Phase extension recommended to operator (advisory only). Operator can override any automatic action.

C8 Multiplicities Single kiln DT with one active drying cycle at a time. No composition of multiple DTs. One validated scenario loaded at a time. Multi-kiln multiplicity identified as a planned extension.
C9 Life-cycle stages Supports the service phase only: execute (batch validation and real-time monitoring), analyse (deviation detection and recommendations), evolve (automatic Ts/damper adjustment), save (sensor history and post-drying report), and terminate (cycle completion or forced stop). No creation phase — DT infrastructure permanently deployed. DT represents the kiln as-operated.
C10 DT models and data Model:
ward_clustering_model: Hierarchical Ward clustering (scikit-learn) used for scenario D batch loading optimization
recipe_model: R40 (EPN/PG, target 16%) and R44 (SAP, target 14%) drying recipes with 7 phases: Heating, Equalization, Auto Steaming 1-3, Fixed Steaming, and Cooling
kiln_3d_model: Godot 4.3 3D scene KilnScene with Package.tscn for lumber lots, rail nodes, fan, coils, door, and GPU particles for steam
drying_physics_model: Exponential MC decay model: dMC/dt = -k*(MC-THE), with K_DRYING constants per species (EPN=0.08, PG=0.07, SAP=0.05)

Data:
sensor_data_streaming: Real-time sensor data published every 2s to wood.kiln.sensor: Ts, Th, derived RH, estimated MC, phase, elapsed hours, fan speed, fan direction, status
history_data: In-memory history of up to 500 data points for dashboard chart reload and post-drying report generation
inventory_data: 70 lumber lots generated by seed.py with species, width, length, and initial MC attributes
C11 Tooling and enablers scikit_learn_enabler: scikit-learn hierarchical Ward clustering for batch loading scenario D
simulator_enabler: Python Simulator.py implementing the 7-phase drying cycle engine, MC decay model, and intelligent control loop
chartjs: Chart.js frontend rendering real-time temperature and MC curves, phase progress bars, and KPI cards at port 8080
fastapi_enabler: Python FastAPI backend exposing REST endpoints for dashboard interaction, cycle control, and scenario management
godot_enabler: — 3D kiln visualization and operator UI for cycle initiation and batch validation
pulp_enabler: PuLP/HiGHS ILP solver for batch loading scenarios A, B, and C
docker: Docker Compose orchestrating Python backend and RabbitMQ containers for reproducible deployment
rabbit_mq: RabbitMQ AMQP broker handling sensor streaming (wood.kiln.sensor), cycle start (wood.kiln.start), and intelligent control commands (wood.kiln.control)
C12 DT constellation runs on the host machine. Communication uses AMQP (RabbitMQ) for sensor streaming (wood.kiln.sensor), cycle start (wood.kiln.start), and intelligent control commands (wood.kiln.control); and HTTP REST (FastAPI) for dashboard and scenario management.
C13 Twinning process and DT evolution Engineered following a POC-first approach for doctoral research. 6 milestones: M1 simulator + RabbitMQ, M2 FastAPI dashboard, M3 Godot 3D scene, M4 batch optimization (greedy to global SSE), M5 deviation detection and recommendations, M6 intelligent control layer. M1-M5: Digital Shadow. M6: full Digital Twin.
C14 Fidelity and validity considerations Models based on established wood science literature. Recipes R40/R44 and deviation rules derived from La Doré industrial practice. MC curves qualitatively consistent with industrial observations. K_DRYING not experimentally calibrated. No CFD airflow modeling. POC not formally validated against La Doré experimental data.
C15 DT technical connection AMQP via RabbitMQ (port 5672): sensor streaming, cycle start, and intelligent control commands. HTTP REST via FastAPI (port 8080): dashboard interaction and cycle control. All communication on localhost and Docker internal network. No physical kiln connection — PT emulated by simulator.
C16 DT hosting/deployment on Windows host; web dashboard at port 8080. Platform-independent — portable to Linux or cloud. No cloud deployment in current POC.
C17 Insights and decision making mc_estimation_result: Real MC (exponential decay) and planned MC (linear reference) updated every 2 real seconds and displayed in dashboard charts and Godot UI
stop_command: Operator-triggered stop via dashboard POST /api/stop; publishes stopped status to wood.kiln.sensor; resets Godot UI to door-open state
post_drying_report: HTML report generated at cycle completion summarizing start/end time, final MC, DTAB statistics, phase durations, and recommendations for the next cycle
monitoring_dashboard: Web dashboard at port 8080 showing MC curves, temperature curves, phase progress, DTAB values, and deviation status
batch_scenario: 4 batch loading scenarios (A: max filling, B: highest quality, C: optimal throughput, D: Ward clustering) with KPIs for operator selection
start_command: Operator-triggered kiln start via Godot UI or dashboard after scenario validation; sends drying_group, width, initial_mc, total_pmp to wood.kiln.start
control_action: Automatic DT control actions published to wood.kiln.control: Ts setpoint adjustment (+-2C) for SLOW_DRYING/FAST_DRYING, damper aperture adjustment for DTAB HIGH/LOW deviations
deviation_warning: Deviation status: ON_TARGET, SLOW_DRYING, FAST_DRYING, CHECK_OVERDRYING; DTAB alerts HIGH/LOW/NORMAL displayed on dashboard
real_time_viz: 3D visualization of kiln state: lumber color by MC, steam particle intensity, fan rotation and direction, coil light energy by Ts, door open/close
phase_extension_recommendation: Advisory recommendation to extend current drying phase when target MC has not been reached; operator confirms or rejects via dashboard
C18 Horizontal integration No horizontal integration with external systems. Standalone DT with synthetically generated inventory. No connection to external systems or other DTs.
C19 Data ownership and privacy Synthetic inventory data — no personal or sensitive information. Drying recipes and rules provided by La Doré industrial partner. MC model parameters informed by proprietary industrial data used within academic context only. No external data transmission.
C20 Standardization Communication via AMQP (RabbitMQ) and JSON payloads. Recipes and parameters follow La Doré industrial norms. No formal compliance with ISO 23247, AAS, or FMI/FMU.
C21 Security and safety considerations No physical safety risk — DT controls simulation only. Local deployment with no external network exposure. Basic isolation via Docker and RabbitMQ — no TLS, no authentication, no RBAC implemented. Industrial deployment would require TLS encryption, authenticated endpoints, and PLC-level safety interlocks.
